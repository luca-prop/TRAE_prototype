가독성	양호 (Good)
변수명(adjustedBudget, showReport)과 컴포넌트명(ComparisonContent)이 명확하며, Tailwind CSS의 클래스명이 직관적입니다. 하지만 comparison/[id]/page.tsx와 같은 파일이 370줄 이상으로 길어져 가독성이 떨어지기 시작했습니다.	컴포넌트 내부의 방대한 렌더링 코드를 기능별로 쪼개기

재사용성	보통 (Fair)
Button, Card, Badge 등 기본 UI는 components/ui/에 잘 분리되어 재사용성이 높습니다. 반면, 데이터가 바인딩된 '특정 도메인 컴포넌트'(기축 아파트 카드, 매물 리스트 아이템)는 페이지 뷰 안에 하드코딩되어 있어 다른 페이지에서 재사용이 불가능합니다.	기축 아파트 카드 UI를 분리
(예: components/domain/ComparisonAptCard.tsx 생성)
유지보수성	보통 (Fair)
페이지 내부 상단에 ZONE_DATA, COMPARISON_APTS, MOCK_PROPERTIES 등 하드코딩된 더미 데이터가 섞여 있습니다. UI 로직과 데이터 로직이 결합되어 있어 추후 실제 API 연동 시 파일 수정 범위가 넓어집니다.	더미 데이터를 외부 파일로 분리
(예: lib/mockData.ts를 생성하고 import { ZONE_DATA } from '@/lib/mockData' 체계화)
일관성	우수 (Excellent)
React/Next.js의 네이밍 규칙(파일: 소문자, 컴포넌트: PascalCase, 변수: camelCase)을 잘 준수하고 있습니다. 또한 디자인 시스템 관점에서도 Tech Blue, Neon Mint 등 정해진 컬러 토큰(blue-600, emerald-500)을 일관되게 사용하고 있습니다.	현재 수준의 일관성(className 작성 순서 및 네이밍 룰) 유지
성능	보통 (Fair)
프로토타입 수준에서는 문제가 없으나, 불필요한 렌더링 방지 처리가 안 되어 있습니다. 예를 들어 렌더링 시마다 formatBudget 같은 유틸 함수나 정렬 로직이 매번 새로 생성/실행됩니다.	1. formatBudget 함수를 컴포넌트 밖으로 빼기
2. 정렬 로직에 useMemo 적용
(예: const sorted = useMemo(() => [...MOCK].sort(...), [MOCK]))
